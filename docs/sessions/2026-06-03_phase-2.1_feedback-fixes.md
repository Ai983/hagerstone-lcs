## What was flagged & what changed

### Q1 — Contractor supplier search returned nothing
- **Diagnosed:** data + permissions fine — `lcs.v_cps_suppliers` returns all 760 suppliers to both `authenticated` and `anon` (tested via `set role`). The bug was the **dual Supabase client**: a separate `createClient({db:{schema:'lcs'}})` sharing the same auth `storageKey` as the default client → supabase-js auth-lock contention → lcs requests hung.
- **Fix:** `src/lib/supabaseLcs.ts` now exports `supabase.schema('lcs')` (single client, single session). Search + all lcs reads/writes now reliably carry the auth token.
- Clarified to user: the picker searches ALL 759 CPS suppliers (to link/enrich), not just contractors.

### Q2 — "Daily-wage labour" meaning (researched + explained)
- Explained the 3 real Indian-construction models (spot/chowk; thekedar/labour-contractor; direct daily-wage) + muster-roll/BOCW context. User chose **B + C**.
- **Migration `lcs_005_workers_labour_mode`:**
  - `contractor_profiles.labour_engagement` text check (`thekedar`|`direct`), nullable (only for type='labour').
  - `lcs.workers` (id, contractor_profile_id FK cascade, name, phone, skill, day_rate, aadhaar_last4, is_active, created_by, timestamps) + updated_at trigger + RLS (read=`is_lcs_user()`; write=ops roles + site_engineer + project_manager) + grants.
- **UI:** contractor form shows a **Labour engagement** select (Thekedar / Direct) when type='labour'. New `WorkersPanel` (expand a labour contractor in the list to add/list workers with day-rate). For thekedar the roster is optional/reference; for direct it's the basis for man-days × rate.

### Q3 — Contractor fields mandatory + auto-fill
- **Auto-fill + lock on link:** linking a CPS supplier fills name/GSTIN/PAN/bank_name/IFSC and makes those read-only. Account number still required (CPS stores only last-4).
- **Mandatory:** name, phone, bank_name, account holder, account number, IFSC — validated before submit.
- **Human tick always required:** the "Bank details verified by a human" checkbox is never auto-set, even when linked (per user decision + the "verify before pay" rule). Sets kyc_status=verified only when ticked.

"Test Project" and already-imported names are filtered out.
